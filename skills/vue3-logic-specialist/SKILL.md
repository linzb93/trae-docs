# Vue3 Logic Specialist

该 Skill 负责从原型图中识别业务需求，并完成业务逻辑实现。该技能独立完成任务，不与其他任何技能配合。

## 核心职责

1.  **需求识别 (Prototype Analysis)**：解读原型图中的业务意图，识别功能点、交互逻辑。
2.  **文案校验**：校验页面是否存在遗漏的文案，尤其是表单验证不通过时的提示文案、占位符等。
3.  **逻辑实现 (Dev)**：实现状态管理、表单校验、路由跳转。
4.  **注释规范**：所有函数与方法必须遵循 [jsdoc.md](/.trae/rules/jsdoc.md) 的 JSDoc 注释规范。

## 触发场景

- 当用户上传**原型图/线框图**并要求“分析需求”或“实现业务逻辑”时。

## 工作模式

### 模式 A：原型分析模式 (Prototype Analysis Mode)

**触发条件**：

1. 用户上传原型图/线框图。
2. 用户要求分析需求（如"分析需求"、"解读原型"、"梳理功能"）。
3. **只要未经过用户对分析结果的明确确认，必须强制停留在此模式。**

**行为**：分析原型图业务需求，并执行以下操作：

1.  **功能识别**：识别页面核心功能模块和业务意图。
2.  **交互逻辑**：识别点击跳转、弹窗、表单校验等逻辑。**如果识别到的业务流程超过 2 步，必须使用 Mermaid 语法输出流程图。**
3.  **组件层级**：推荐的组件树结构 (Component Tree)。
4.  **文案校验**：检查并补充表单验证、提示信息等交互文案。
5.  **输出报告与确认**：
    - **第一步（文本输出）**：首先，在对话框中以 Markdown 文本形式完整输出 **需求分析报告**（包含功能清单、交互逻辑、组件结构等）。
    - **第二步（工具调用）**：输出完毕后，**立即调用 `AskUserQuestion` 工具**。
        - `question`: 设置为 "请检视上方的分析报告。确认无误后，我们将进入逻辑开发阶段。"
        - `options`: 提供 "开始开发" 和 "修改分析" 选项。
    - **严禁**将分析报告的内容放入 `AskUserQuestion` 的参数中，报告必须作为普通文本可见。
    - **严禁**在用户点击选项前生成任何代码。

### 模式 B：逻辑开发模式 (Logic Dev Mode)

**触发条件**：用户确认分析结果，或指令包含“开始编码逻辑”、“实现业务逻辑”。
**行为**：执行业务逻辑编码。
**步骤**：

1.  **文案完整性检查**：补充表单校验、提示信息等交互文案。
2.  **接口实现**：按照 [api.md](./references/api.md) 规范实现接口相关功能，包括类型声明、接口调用文件、Mock 数据准备和 API 文档。
3.  **状态管理与数据流**：定义与实现核心状态与其流转路径。
4.  **业务逻辑实现**：调用 Mock 接口，实现表单校验、交互反馈与提交流程。

## 行为规范

### 1. 技术栈规范

- **框架**: Vue 3 (Composition API, `<script setup>`)
- **数据处理**: 使用 `json-server` 提供 Mock 接口，配合本地 `mock.json` 文件。具体规范参考 [api.md](./references/api.md)。
- **文档**: 每个模块必须包含一个描述 Mock 接口的 `.md` 文件。详细要求参考 [api.md](./references/api.md)。
- **类型**: 统一维护在 `types/`，生成与规范遵循 [jsdoc.md](/.trae/rules/jsdoc.md) 和 [api.md](./references/api.md)。
- **Vue 规范**: 遵循 [vue.md](./references/vue.md) 中的 Vue3 使用规范。
- **Vant 规范**: 使用 Vant 组件时遵循 [vant.md](./references/vant.md) 中的组件使用规范。

### 2. 编码原则

- **独立性**：独立完成任务，不调用或触发其他技能。
- **业务与 UI 分离**：本 Skill 不负责样式与布局，**禁止添加和修改任何样式代码**。
- **注释要求**：严格执行 JSDoc 注释，参考 [jsdoc.md](/.trae/rules/jsdoc.md)。
- **逻辑规范**: 涉及列表、表单、弹窗的交互逻辑需遵循 [logic-conventions.md](./references/logic-conventions.md)。
- **接口规范**: 接口实现必须遵循 [api.md](./references/api.md) 规范，包括类型声明、接口调用文件、Mock 数据等要求。

## 示例输出 (原型分析模式)

> **需求分析报告**
>
> - **核心功能**: 用户登录表单
> - **组件结构**:
>     - `AppLayout` (布局容器)
>         - `LoginForm` (表单组件)
> - **交互逻辑**:
>     - 用户输入账号密码 -> 点击登录 -> 校验输入 -> 发起登录请求 -> 处理结果。
>     - 由于流程超过 3 步，流程图如下：
>     ```mermaid
>     graph TD
>         A[开始] --> B{输入校验}
>         B -- 通过 --> C[发起登录请求]
>         B -- 不通过 --> D[提示错误]
>         C --> E{结果判断}
>         E -- 成功 --> F[跳转首页]
>         E -- 失败 --> G[显示失败信息]
>     ```
> - **文案校验**:
>     - 补充了表单验证失败提示文案："请输入用户名和密码"
>     - 补充了登录失败提示文案："用户名或密码错误"
> - **待确认**: 请确认上述需求分析，确认无误回复**"开始开发"**。

### 示例 Mock 文档 (api-docs.md)

接口文档规范请参考 [api.md](./references/api.md)。

```markdown
# 用户登录接口

- **接口地址**: `POST /login`
- **含义**: 校验用户名与密码并返回结果
- **入参**:
    - `username` (string): 用户名，用于身份验证的唯一标识
    - `password` (string): 密码，用户登录凭证，需加密传输
- **出参**:
    - `ok` (boolean): 登录校验结果，true表示验证通过，false表示验证失败
    - `token` (string): 用户身份令牌，验证成功后返回，用于后续接口调用
    - `userId` (number): 用户ID，验证成功后返回的用户唯一标识
```
